# LRP02 quantile α=0.5 objective — direct median optimisation

> [!WARNING]
> These notes were generated by an AI tool.

Date: 2026-04-17

## Motivation

After the LRP02 log-transform + cap-fix work, the significant result
on this task is `lrp02` → `lrp02_log` MedAE (p=0.005, 9/10 folds).
That improvement arises because:

1. The MAE objective (`objective="mae"` in LightGBM) gives uniform
   ±1 gradient regardless of residual size, so the model optimises
   toward the **conditional median** of the target given features.
2. MedAE directly measures the absolute error at that conditional
   median.
3. Log-transforming the target reshapes the residual distribution,
   which pushes the conditional median closer to the actual
   median observation.

Natural follow-up: LightGBM also supports `objective="quantile"`
with `alpha=0.5`, which minimises the **pinball loss at the
median** — the direct median-optimising objective. In theory MAE
and quantile(α=0.5) converge to the same point estimator, but
LightGBM's gradient handling differs between `mae` (±1 uniform)
and `quantile` (±α vs ±(1−α), which at α=0.5 is ±0.5), and the
Hessian approximation differs too. The empirical question is
whether the quantile objective produces a measurably different
CV MedAE.

This PR adds two quantile variants and compares them head-to-head
against the MAE counterparts.

## Implementation

### `scripts/tune_model.py` — `--alpha` flag

Added an `--alpha` CLI argument to `tune_model.py`. LightGBM needs
`alpha` when `objective` is `quantile` or `huber`; the tuner now
accepts it and threads it through the search space:

```python
if alpha is not None:
    params["alpha"] = alpha
```

Also added `medae` to `_SCORING_FUNCS` so the tuner can directly
optimise MedAE (which is the natural scoring metric for the
quantile(0.5) objective).

Usage:

```bash
python scripts/tune_model.py lrp02_quantile \
    --n-trials 150 --scoring medae \
    --lgbm-objective quantile --alpha 0.5
```

The `target_transform` auto-infer (`LGBMPipeline → none`,
`LGBMLogPipeline → log1p`) still applies.

### Variant classes

Two new classes in `models/lrp02.py`:

- **`LRP02Quantile(LRP02)`** — same 13-predictor feature set as the
  primary, `pipeline_cls=LGBMPipeline`, `objective="quantile"` at
  `alpha=0.5`.
- **`LRP02LogQuantile(LRP02Log)`** — same 13-predictor feature set as
  `lrp02_log`, `pipeline_cls=LGBMLogPipeline`,
  `objective="quantile"` at `alpha=0.5`, log1p target transform.

Both get dedicated `_LGBM_QUANTILE_PARAMS` and
`_LGBM_QUANTILE_PARAMS_LOG` dicts from Optuna 150-trial tunes
(scoring=medae, lgbm_objective=quantile, alpha=0.5).

Both tunes ran on 13-predictor Select04 set, 10-split GroupKFold,
seed 47, scoring=medae, 150 Optuna trials.

| `lrp02_log_quantile` | log1p | **2.896 ± 0.771** | 322 | 7 | 32 |

Both are meaningfully better than the MAE-tuned counterparts'

### Observations

- **Raw target: quantile improves EVERYTHING.** CV MAE −0.21,
  CV R² +0.08, CV MedAE −0.31, CV R² std 0.41 → 0.32. In-sample R²
  unchanged (0.919) so no extra over-fit. If you are stuck on the
  raw target, quantile α=0.5 dominates MAE.
- **Log target: quantile is a wash.** All metrics basically tied
  with `lrp02_log`. On a log-compressed target the residual
  distribution is already well-conditioned, so the MAE vs
  quantile(0.5) gradient-handling differences don't matter.
- **`lrp02_log` remains the overall best** on CV R² and CV MedAE
  across all four configurations.

### Quartile in-sample MAE

| Target range | `lrp02` | `lrp02_quantile` | `lrp02_log` | `lrp02_log_quantile` |
|---|---|---|---|---|
| (0, 1] | 0.45 | 0.94 | 0.32 | 0.40 |
| (1, 6.5] | 0.43 | 0.88 | 0.30 | 0.39 |
| (6.5, 17] | 1.18 | 1.35 | 0.78 | 0.70 |
| (17, 64] | 2.72 | 5.70 | 2.63 | 3.57 |

The raw-target quantile variant (`lrp02_quantile`) has *worse*
in-sample MAE in the low-value quartiles (0.94 / 0.88 vs lrp02's
0.45 / 0.43) but compensates via much better generalisation —
recall that `lrp02`'s in-sample MAE of 0.45 was largely
memorisation (in-sample R² 0.919 → CV R² 0.314 gap), while
`lrp02_quantile` spreads capacity more broadly.

The log-target quantile variant (`lrp02_log_quantile`) closely
mirrors `lrp02_log` across quartiles with slight degradation on
the top quartile (3.57 vs 2.63). This is where the tiny R²
regression (0.489 → 0.462) comes from.

## Paired-fold tests (α=0.05 significant results)

Full 4-way `compare_variants.py` output shows only these pairs
reach significance:

| Comparison | Metric | mean diff | paired *t* p | Wilcoxon p | B wins |
|---|---|---:|---:|---:|---:|
| `lrp02` → `lrp02_log` | MedAE | −0.877 | **0.005** | **0.006** | **9/10** |
| `lrp02` → `lrp02_log_quantile` | MedAE | −0.824 | **0.038** | **0.037** | 8/10 |
| `lrp02_quantile` → `lrp02_log` | MedAE | −0.562 | **0.036** | **0.037** | 8/10 |

Every significance is on **MedAE** — the metric the quantile
objective directly optimises. Every significant comparison
involves the **log transform** — either moving to a log variant
or away from one.

**`lrp02` → `lrp02_quantile`** (changing the objective only, raw
target both sides) does NOT reach significance on any metric
(MedAE t p=0.222, 5/10 wins; MAE t p=0.345, 7/10 wins). The
improvements are directional and consistent but within fold
noise at n=10.

**`lrp02_log` → `lrp02_log_quantile`** (changing objective only,
log target both sides) is essentially tied on every metric (t p
> 0.2 across the board, 4/10 wins on most metrics).

## Interpretation

1. **The log transform is the dominant lever.** Every significant
   MedAE improvement involves moving to a log-target variant.
2. **Quantile objective matters only on the raw target.** When
   LightGBM is training on raw `ewrswr` with its long right tail,
   MAE's uniform gradient leaves the conditional median slightly
   misspecified; quantile(0.5) fixes that. The effect is
   directional but doesn't reach α=0.05 at n=10 folds.
3. **On a log-compressed target, MAE and quantile(0.5) are
   equivalent in practice.** The tail is already tamed by the
   transform, so the gradient-handling difference washes out.
4. **`lrp02_log` remains recommended.** The best-performing
   combination on CV MedAE across everything tried is still
   log-transform + MAE objective. Adding the quantile objective
   on top of the log transform provides no additional benefit.
