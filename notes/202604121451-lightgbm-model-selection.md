The four candidates lie within 0.04 RMSE units of each other — well inside one cross-fold standard deviation. Accuracy alone cannot pick a winner. Speed becomes the deciding criterion.

## Speed test

End-to-end fit on `--config reporting` (53-split GroupKFold + permutation importance + SHAP TreeExplainer + partial dependence plots), measured on 2026-04-12 from `dev/tune-lrp01-rf`:

| Model | Wall time |
|---|---|
| `lrp01_select01` (tuned RF, 1800 trees) | **7 min 11 s** |
| `lrp01_lgbm_select01` (tuned LGBM, 83 rounds) | **14 s** |

**LightGBM is ~30× faster end-to-end** for an essentially identical CV RMSE. At this margin the choice is not even close.

## Why LightGBM is so much faster — the full picture

The 30× gap is the product of several independent multipliers, not a single architectural trick. Each of the following contributes:

### 1. Tree count (≈ 21.7× fewer trees)

The tuned RF uses `n_estimators=1800`. The tuned LGBM uses `n_estimators=83` (chosen by per-fold early stopping, then taking the mean best iteration across folds). Every downstream cost — fit, prediction, permutation importance, SHAP, PDP — scales with the number of trees. Before any per-tree differences kick in, LGBM is already doing 21.7× less work just because there are fewer trees.

This is not an accident of tuning. **Random Forest needs many trees because each tree must independently learn the full input-output mapping**, and the ensemble averages away their variance. **Gradient boosting needs few trees because each tree only fits the residual of the previous trees** — they cooperate rather than vote, and a shallow corrective tree on the residual is enough.

### 2. Split-finding: histograms vs exact scans

This is the biggest *per-tree* difference and the reason LightGBM (and XGBoost's `hist` mode) revolutionised GBDT speed in 2017.

**Random Forest (sklearn `RandomForestRegressor`)** uses *exact* split finding. For each candidate split at each node, it must:
- For each of the considered features (here `max_features=0.375`, so ≈13 of 34 features per node)
- Sort the samples by that feature's value
- Scan every unique value as a potential split threshold
- Compute the loss reduction at each threshold

This is O(features × samples × log samples) per node. With n=152 it's not catastrophic, but it happens at every internal node of every one of the 1800 trees, with `max_depth=11` allowing up to 2¹¹ = 2048 leaves per tree (`min_samples_leaf=16` caps this in practice but the search still happens).

**LightGBM** uses *histogram-based* split finding. Once at the start of each tree (and incrementally as the tree grows):
- Each continuous feature is bucketed into ≈255 fixed bins
- For each feature × bin combination it accumulates the gradient and hessian sums
- Split finding becomes a sequential scan over **bins**, not samples — O(features × bins) per node, completely independent of n

For our 152 samples × 34 features the bin count (255) is actually larger than the sample count, so the absolute speedup at this n is modest. **But LightGBM also uses the histogram subtraction trick**: once the parent's histogram and one child's histogram are computed, the other child is `parent − child`, halving the histogram-building work at every level. Combined with cache-friendly sequential bin scans (vs RF's pointer-chasing through sorted sample indices), the per-node cost is several times lower even at small n.

### 3. Tree growth strategy: leaf-wise vs level-wise

**Random Forest** grows each tree level-by-level (depth-first within sklearn's implementation, but the *shape* is constrained by `max_depth`). To reach `max_depth=11` it grows the full level-wise structure, even branches that wouldn't have meaningfully reduced the loss.

**LightGBM** grows leaf-wise: at each step it expands the leaf whose split would reduce loss the most. Because each tree is small (`num_leaves=34`, `max_depth=12`), it finds deep structure where it matters and stops growing in directions that don't help. This produces trees with ≤34 leaves rather than RF's potential thousands of leaves at depth 11. Each tree is doing dramatically less work — and there are 21.7× fewer of them.

### 4. Per-tree training-set size

The tuned RF has `bootstrap=False`, which means **every one of the 1800 trees fits all 152 samples** — there's no bootstrap subsampling to reduce per-tree cost. (Optuna picked `bootstrap=False` because at n=152 the variance reduction from bootstrapping isn't worth the loss of training signal.)

LightGBM with `subsample=0.836` shows each tree only ≈127 samples. Combined with leaf-wise growth, each LGBM tree is doing a fraction of the work of an RF tree even before you count the histogram speedup.

### 5. SHAP TreeExplainer cost

`TreeExplainer` computes exact Shapley values via the Lundberg et al. 2018 algorithm. The cost is approximately **O(T × L × D²)** per explained sample, where T = tree count, L = leaves per tree, D = max depth.

- **RF**: 1800 trees × ~hundreds of leaves × 11² ≈ enormous
- **LGBM**: 83 trees × 34 leaves × 12² ≈ trivial

For our small dataset (152 samples to explain, all of them), this single step accounts for a substantial chunk of the RF wall-time. SHAP on the LGBM model is essentially free.

### 6. Permutation importance cost

`sklearn.inspection.permutation_importance` with `n_repeats=10` (the default in `EstimatorPipeline`) makes **10 × n_features = 340 full predictions through the model** per call, on top of the baseline. Each prediction walks every tree.

- RF: 340 × 1800 trees × ~depth-11 traversal
- LGBM: 340 × 83 trees × ~depth-12 traversal

A ~21× ratio just from tree count, before considering that LGBM's binned trees have faster traversal due to better cache behaviour.

### 7. Cross-validation amplification

The reporting config uses 53-split GroupKFold (leave-one-subject-out, since there are 53 unique subjects). Every cost above is paid **53 times** during cross-validation, then once more on the full-data refit. The constant factor matters.

### 8. Native parallelism

Both estimators run with `n_jobs=16`. RF parallelises over **trees** — each worker fits one independent tree. LGBM parallelises over **feature histograms** within a single tree. At small n with bin counts > sample counts, neither parallelism path is fully saturated, but LGBM's data layout is more cache-friendly so its threads spend less time waiting on memory.

## The compounding effect

These factors do not add — they multiply.

```
Speedup ≈ (tree count ratio) × (per-tree work ratio) × (SHAP/PI tree-count amplification)
        ≈ 21.7×                × ~3-5×                × already counted in tree count
        ≈ 30×  end-to-end
```

The fit and CV phases dominate the RF wall-time; SHAP and permutation importance dominate the residual. LGBM compresses every one of them.

## Operational consequences

1. **`lrp01_lgbm_select01` becomes the default `lrp01` model.** Plan: rename it to plain `lrp01` (or alias the unsuffixed entry to its config) and demote the RF entries to `variant_of="lrp01"` so they're skipped by `fit_model.py all` but can still be reproduced explicitly.

2. **Future tuning effort focuses on LGBM only.** The RF symmetric tuning in PR #3 was worth doing for the methodological comparison, but going forward the Optuna budget belongs to LGBM.

3. **Same logic should apply to lrp02** when it gets the same treatment — no need to maintain two algorithm families per problem.

4. **Faster iteration on feature selection.** With 14 s end-to-end fits, the `analyze_predictors.py → drop a feature → fit_model.py lrp01 → compare metrics.json` loop becomes interactive. The 7-minute RF cycle made selection rounds painful enough to discourage exploration.

5. **Bespoke template comparison sections still work.** `docs/models/lrp01.qmd` already groups by estimator family; if/when the RF variants are removed entirely the LGBM family section just becomes the only one.

## What to keep from the RF work

- **PR #3's tuning script changes are general** (Optuna RF objective, `--early-stopping-fraction` flag for LGBM, raw-NaN data loading) — they stay regardless.
- **The RF tuned config is documented in `lrp01.py`** for the historical record. Even if we stop fitting it by default, the variant entry preserves the parameters and the comparison.
- **The methodological lessons** (leak fix, NaN unification, symmetric inner-CV honesty) apply equally to LGBM. They were the substance of PR #2; PR #3 was the validation that they generalise.

## Caveats

- These timings are at n=152. The histogram speedup grows with n; at n=10,000 the gap would be far larger. At n<100 it would shrink. Future LRP cohorts only push this further in LGBM's favour.
- LGBM's 14 s includes some fixed Python startup, conda activation, data load — the *marginal* fit cost is even lower. Re-fitting in a notebook session where the import cost is amortised would be sub-second.
- The accuracy tie is at this specific predictor set. If feature selection meaningfully changes the predictor count, the gap could re-open in either direction. Worth re-checking after the next selection round.

### Agreement

The same eight predictors fill the top eight in both models — `celf`, `attend`, `age`, `eowpvt`, `b1exto`, `blending`, `yarclet`, `time`. The clinical story (intervention attendance + age + receptive language + early reading skills) is identical across estimators. This is reassuring: the SHAP rankings are not an artefact of the algorithm, they reflect signal in the data.

### Differences worth noting

1. **Magnitudes are larger for LGBM.** LGBM concentrates ~0.7 on the top two features vs RF's ~0.45. Boosted trees pile signal onto a few strong splits because each successive tree only adds where the previous trees left residual; RF's 1800 independent trees average importance more diffusely across redundant predictors.

2. **`celf` drops from #1 (RF) to #3 (LGBM)** while `attend` and `age` rise. RF's exact split-finding over n=152 finds more `celf` cut-points to exploit; LGBM's histogram bins coarsen `celf` and let `attend`/`age` dominate the top of the tree.

3. **`time` jumps** from #8 (RF, 0.08) to #5 (LGBM, 0.31). LGBM is finding a stronger interaction or non-linear `time` effect that RF averages out across its bagged ensemble.

4. **The long tail differs sharply.** LGBM's permutation importance assigns **exact zero** to seven features (`group`, `aptgram`, `area`, `spphon`, `deappin`, `behav`, `vision`) — leaf-wise growth combined with `reg_alpha=1.49` simply never splits on them. RF assigns every feature a tiny non-zero value because some tree, somewhere, finds use for it.

5. **`deappin` is the most striking single disagreement** — RF ranks it #10 in permutation importance; LGBM zeros it. Worth checking in the next feature-selection round whether it is redundant with another feature in the same correlation cluster.

### Implication for feature selection

For the upcoming `analyze_predictors.py` → drop → refit loop, **LGBM is actually more helpful than RF** even setting aside the speed gap. The explicit zero-importance features are a free first-pass drop list — no threshold call required. RF's "everything is slightly important" output forces an arbitrary cutoff. Combined with the 30× speed advantage, the LGBM-driven selection cycle should be both faster and cleaner.

### Caveat on comparing magnitudes

The absolute SHAP magnitudes between RF and LGBM are not directly comparable — they sum to (predicted − base value) for each model, and the two models partition the same predicted variance differently. The *ranks* and the *concentration pattern* are the meaningful comparison; the raw numbers are not.
